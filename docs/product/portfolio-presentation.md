# Contäx — ポートフォリオ プレゼン整理

> UIUXのプレゼンフロー（7ステップ）に沿って内容を言語化したもの。
> 話し言葉への変換・スライド構成は別途行う。

---

## 1. 文脈の設定

**課題発見までのプロセス（スライドの4ステップ構成）**

> AI時代において「書き終える」ではなく「自分の言葉で伝える」ためのツールが、まだない

1. **時代背景**：AIの登場により、誰でも早く綺麗な文章が作成可能になった。ChatGPTを筆頭に、それらしい文章が一瞬で生成できる時代になった。
2. **ヒアリング（友人）**：適当に利用すると質が下がる。友人もそのような声が多く、出力をそのまま使うことへの違和感を自覚していた。
3. **ヒアリング（教員）**：「AI使用の文章でも差がつく。裏付けられた経験があるかどうかだ」——教員視点でも差がついて見えることが判明。
4. **課題の仮説**：問題はAIを使う・使わないではなく、**AIと人の役割分担が整理されていないこと**。人がやるべきプロセスを踏みながらAIを効果的に活用することがこの先必要。

- Contäxは「書き終える」ではなく「自分の言葉で伝えられる」状態をゴールに置いたツール

**スコープ**
- プロジェクトのゴールはMVP開発まで
- MVP範囲：①エディター画面で文章が書けること、②経験を登録してヒントとして使えること

## 2. リサーチ

**調査手法**
- SNS（X）・既存ツールレビュー（Notion AI / Catchy 等）・掲示板（Reddit / Yahoo!知恵袋）のデスクリサーチ

**ファインディング（3つ）**

| # | 発見 | データ |
|---|------|--------|
| 1 | ESのAI利用率は顕著に | 就活でAIを利用している **83%**（デスクリサーチ） |
| 2 | 一方で懐疑的な意見も | 「自分で考えていないので使わない」 **28%**（デスクリサーチ） |
| 3 | 面接で言葉に詰まる | 自分の言葉で書いていないことが原因（デスクリサーチ） |

- AIに丸投げすると「書いた内容を面接で答えられない」という状態に直結している
- 「丸投げへの後ろめたさ」を多くのユーザーが自覚している = 解決策を求める下地がすでにある

**ニーズの構造**

| レイヤー | 内容 |
|---|---|
| 表面 | 綺麗な文章を書きたい・AIにバレたくない |
| 中間 | 自分らしい文章を書きたい |
| 深層 | 自己言語化できている状態になりたい |
| **本質** | **書いたことを自分の言葉として語れる・所有できる状態（思考の所有権）** |

**現状と競合（競合マップ）**

2軸で競合をマッピング：
- X軸：違和感（AIに頼りっきり） ←→ 伝達（自分の言葉で伝えられる）
- Y軸：効率（速く・楽に書く）←→ 伴走（人が考えるプロセスを支援）

| ツール | 位置 |
|---|---|
| ChatGPT / Catchy | 違和感 × 効率 |
| Gemini / Claude | 違和感寄り × 効率 |
| Notion | 伝達寄り × 効率 |
| **Contäx** | **伝達 × 伴走（右上・空白地帯）** |

- 既存ツールは「速く・楽に書く」に集中しており、市場の大半が「効率 × 代行」象限に集まっている
- ContäxはAIフル生成フローを意図的に持たない「プロセスの設計」で差別化している

## 3. 問題の定義

**HMW（解くべき問い）**
> どうすれば、AIを使いたいが丸投げへの後ろめたさを感じている大学生が、自分の思考・経験をプロセスとして整理しながら、AIを正しく活用した文章を作れるようになるか？

**プロダクトが解く2つの問題**

| # | 問題 | Contäxの解決策 |
|---|------|----------------|
| 1 | 構成を自分で考えずAIに渡してしまう | 構成を自分で組むプロセスをUIに組み込む |
| 2 | 伝えたいこと・経験・学びが整理されないまま出力される | それらを言語化・記録するステップをフローに残す |

ポイント：AIを「使わない」のではなく、「人がやるべき部分を先にやってからAIに渡す」順序を設計している。

## 4. 設計の探索

- 解決策の候補は①構成を自分で組む、②思考・経験をプロセスとして残す、の2軸に早い段階で絞られた
- 問題定義が明確だったため、探索の幅よりも「どう実装するか」の検討に移行した
- 初期UIは紫グラデーション → 「AIツール感が強すぎる・ChatGPTとの差別化を損ねる」という課題が出てきたため再検討へ

## 5. 意思決定

### 画面構成の判断
- 「構成を組む」と「経験を掘り起こす」は**別画面に分けた**
- 理由：構成を組む場面で、自分の経験がここでマッチすると気づくことが最適なタイミング。経験を掘り起こすプロセスを同じ画面に混在させると、ユーザーの認知の順序を崩す

### PREP法をデフォルト構成に設定した判断
- 試行錯誤で疲弊しているユーザーが最初に開いたとき「どう構成するか」まで考えさせない
- PREP法をデフォルトにすることで「何を書くか」だけに集中できる状態を作る
- ただしツールに慣れてきたら自分の型を保存・再利用できる設計にしている
- 初心者はデフォルト型に頼り、使いこなすにつれて自分の型を持てるようになる → Contäxのコアバリュー「思考の所有権・スキルを育てる」と対応している

### AIフル生成機能を入れない判断
- AIが全部書く機能は意図的に除外した
- 理由：そこがわざわざContäxを使う差別化の核心だから。AIに全部書かせるならChatGPTで十分

### ブロック型エディタの選択
- 量を書くことが苦じゃなくなった時代、差を生むのは構造
- AI活用が上手い人の使い方を観察すると、箇条書き・構造化で情報の優劣やまとまりを可視化している
- ブロック型エディタはその「構成の可視化」に最も適した形式だった
- フリーテキストでは構成が埋もれる

### エディター画面のUI設計判断

#### ヒントを「全体（左パネル）」と「ブロック個別（インライン）」に分けた判断

**経緯：**
- 当初はヒントとコンテンツを同一ブロック内に配置していた
- 実装・検証した結果、ヒントを見るためにスクロールするとセクションブロックも連動して動き、UXストレスが発生することが判明
- 各ブロックの長さがバラバラになることでこの問題が顕在化した

**設計の結論：スクロール領域を分離し、2層構造にした**
- **左パネル（マクロ）**：全体ヒント＋使えそうな経験の提示。構成全体を俯瞰しながら常時参照できる
- **インライン（ミクロ）**：各ブロックごとの具体的な書き方ヒント。書いている箇所のすぐそばで参照できる

**なぜこの構成か：**
- 「全体構成を俯瞰しながら、各ブロックに集中して書く」というフローを画面構造として実現したかった
- Contäxのコアバリュー「構成を自分で組みながら書く」と直結している——左で全体を見ながら右で具体を書く、という俯瞰と集中の往復を設計に組み込んだ
- スマホではスクロール分離の技術的制約により左パネルは折り畳み式になるが、インライン（ミクロ）は残るため書く動線は保たれる。失われるのは全体俯瞰と経験提案のみ

#### ラベルを問いかけ形式にした判断
- 「結論・理由・経験」という名詞ラベルではなく、文章の接続が見えやすい問いかけ・動詞形式にした
- 直感的な判断だったが、UXライティングの観点でも動詞・問いかけ形式のラベルは名詞形より行動誘導効果が高いとされており、原則と合致していた
- 意図的な検証に基づく判断ではなかったため、効果の計測は今後の課題

#### フローティングバーを採用した判断
- フィードバックで「スクロールしていない」という声が複数あり、下にブロックが続いていることが伝わっていなかった
- フローティングバーを導入することで「この下にもエディターが続いている」ことを視覚的に示した
- ユーザーフィードバック → 問題の仮説 → UI改修という検証ループの事例

---

### ターゲットの更新（重要な意思決定）
- **当初の想定**：AIに抵抗を持つ人。使うこと自体へのハードルを下げることを優先
- **リサーチ後の更新**：「丸投げへの後ろめたさ」を感じながら使っている層の方が、構成を組む・経験を言語化するというContäxの2つの解決策に対してより力を発揮できるという仮説が生まれた
- **根拠**：
  - すでにAIを使うモチベーションがある → ツールを使い始める動機が自然に生まれる
  - 「後ろめたさの自覚」= 解決策を求める下地がすでにある
  - Contäxの設計（構成→経験言語化→AI活用）が直接刺さる
- **結論**：メインターゲットをAI活用の質を上げたい層に重心を移した。AI抵抗層も使えるが副次的なユーザーとして位置づける

## カスタマージャーニー

**メインパス**

| フェーズ | 行動 | 感情 | 課題 |
|---|---|---|---|
| **①気づき** | 就活開始・ESを書く必要が出てくる | 焦り・やらなきゃ | 何をどう書けばいいかわからない |
| **②試行** | ChatGPTでESを書いてみる | 意外と簡単・これでいいか | 出力はされるが自分で考えていない |
| **③違和感** | 出力に納得できない・自分の言葉じゃない気がする | モヤモヤ・後ろめたさ | 何が違うのかわからない |
| **④試行錯誤** | プロンプトを変える・書き直す・別ツールを試す | 疲弊・どうすればいいんだろう | 試してもうまく書けない（大多数がここで詰まる・リサーチで裏付けあり） |
| **⑤出会い** | Contäxを知る | 期待・半信半疑 | — |
| **⑥使用** | 構成を自分で組み・経験を言語化してからAIに渡す | 少し面倒・でも納得感がある | オンボーディングがないと迷う |
| **⑦所有** | 書いた内容を自分の言葉として語れる | 自信・スッキリ | — |

**サブパス（遅延トリガー）**
③違和感を感じつつもChatGPTを使い続ける → ESは通過する → 面接で内容を聞かれ答えられない → 危機感からContäxへ

**Contäxの介入ポイント**
④試行錯誤で詰まっている段階。「うまく書けないのはAIの使い方ではなく、自分の思考を整理するプロセスが抜けているから」という気づきを提供する。

## 6. 検証

**方法**：ベータ版を公開しユーザーフィードバックを収集。エンジニア・デザイナー双方の視点で課題を整理。

**コアバリューとして確認できたこと（変えない部分）**
- 文字数ゲージによる進捗の可視化 → 「文章のものさし」として機能している
- 構成ブロックによる執筆支援 → プロンプトのラリーを減らす体験が有効
- AI出力を直接編集してWord出力まで完結する流れ → 実用性が評価された

**発見した問題とBefore/After**

| 問題 | 改善 |
|---|---|
| 別タブ移動でAIヒントが消える | sessionStorageと同期し状態を永続化 → 解決済み |
| 文字数オーバー時に結論ブロックが削れる | プロンプトに文字数制約を厳密に指示するロジックを改修 → 解決済み |

**デザイン視点で発見した課題（未解決・次フェーズ）**
- スマホ版でヒントブロックがスクロールを阻害する → PC「2画面分割」をスマホ「タブ切り替え」に変更が必要
- 経験管理画面が暫定UIのまま → 自動蓄積と手動登録を整理できるUIに刷新が必要
- 初回ユーザー向けオンボーディングが未実装 → ウォークスルー・ダイアログの設計が必要

**仮説のまま（未検証）**
- 「計測ツール」へのブランディングシフトがターゲットに刺さるか
- インポート機能（ChatGPTからの移行支援）のニーズがあるか

## 7. 振り返り

**発見できたこと**
- オンボーディングが足りておらず、ユーザーがコア機能にたどり着けていなかった
- 「普段使っているLLMからの乗り換えが大変」という意見があった（スイッチングコストの存在）
- ターゲット層（AIへの解像度がそこまで高くない層）は、アプリに対して「どんどん試す」前提がない。そのためフィードバックを取りに行かないと気づけない種類の課題がある

**学んだこと**
- ターゲットがデジタルツールに慣れていない層の場合、受動的なフィードバック収集では不十分。能動的にテストセッションを設けて観察する必要があった
- 機能が揃っていても「入口の設計」がなければ意味がない。オンボーディングは後付けではなく、設計の初期から組み込むべきだった

**次にやること**
- オンボーディングのウォークスルー設計（文字数設定→テーマ設定→生成の流れを直感的に）
- スマホのレスポンシブUI刷新（ヒントブロックのタブ切り替え対応）
- 経験管理画面の正式デザイン

**チームでの対立・意思決定**

- **構成自動生成機能をめぐる対立**：友人ユーザーから「構成の型がわからない」と言われ、解決策として構成の自動生成機能を実装した。しかし途中から参加したチームメンバーに「コンセプトとズレている」と指摘される。「ペルソナのゴールは自分で構成を考えること、AIはあくまで効率化の手段。自動で組んでしまうとそのスキルが養われない」と自分でも矛盾を確認し、構成のヒントは生成するが組むのはユーザー自身、という設計に変更した。

**もっとこうできた**
- 「AIがヒントに対して最適な構成を提案しワンクリックで型を組める」機能は、「面倒さを下げる」という意図で作った。しかしContäxのコアバリューである「人が構成を自分で組む」と矛盾していた。ユーザーがAIの提案をそのまま適用するだけでは、ChatGPTの出力をそのまま使うことと構造的に変わらない。提案された構成に対してユーザー自身が判断・編集する余地を設計に組み込むべきだった。
- テンプレート画面に「今のエディタ構成から作成」という動線を実装したが、これはコアバリュー「PREP法に頼る初心者が、使いこなすにつれて自分の型を持てる」と後から気づいたら合致していた。意図せず合致していたのであって、設計段階でコアバリューとの接続を意識していれば、オンボーディングとの連動やUIコピーもその成長フローに合わせて設計できたはずだった。
- 経験データ画面が暫定UIのまま公開されたことで、経験蓄積のUXが粗く使われにくい状態になった。経験データは左パネルの「使えそうな経験の提案」の源泉であり、ここが機能しないとコアバリューの一部が届かない。スイッチングコストへの回答として想定していた自動抽出（外部LLMエクスポートのインポート機能）も未実装のままで、乗り換えの障壁を解消できていない。

## ブラッシュアップタスク（次フェーズ）

| 優先度 | タスク | 理由 |
|--------|--------|------|
| 高 | オンボーディングのウォークスルー設計 | コア機能にたどり着けていないユーザーが多数。入口の設計がなければ機能が届かない |
| 高 | 経験管理画面の正式デザイン | 暫定UIで使われにくく、左パネルの提案機能が活きていない |
| 高 | 経験の自動抽出（外部LLMインポート）機能の実装 | スイッチングコストへの回答。未実装のため乗り換え障壁が残ったまま |
| 中 | スマホのレスポンシブUI刷新（ヒントブロックのタブ切り替え対応） | スマホでのスクロール阻害が未解決 |
| 中 | テンプレート画面のUIコピー・導線をコアバリューと接続して再設計 | 「PREP法 → 自分の型」の成長フローが画面上で伝わっていない |
| 低 | 「計測ツール」へのブランディングシフトの検証 | ターゲットに刺さるか未検証 |

## チーム体制

- 2人体制：りく（UI/UXデザイン・フィードバック改善）、もう1人（開発）
- 実質的に個人開発に近い構成。デザイン判断はほぼりくの裁量
- デザインと開発の役割が明確に分かれているため、「なぜこの設計にしたか」の意思決定はデザイナーとして個人視点で語れる
